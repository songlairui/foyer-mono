## 起意

用户本机的 Git 仓库散落在多个目录：`~/repo`、`~/repos`、`~/github`，以及已由 foyer 管理的 `~/repo/projects`（含嵌套结构）。现有 `foyer repo devices` 仅扫描单一的 `projectsRoot`（默认 `~/repo/projects/`），导致其他路径下的 repo 对 foyer 完全不可见，跨设备同步时也无法感知这些"游离"仓库。

tap 提出三个相互关联的问题：多目录如何编排、跨设备是否同步、是否引入 playground 阶段区分"探索中"与"落户"仓库。

当前 `repo.ts` 的已有实现：

- `repoDevices`（第 13 行）：扫描单一 `config.projectsRoot`，返回含 `.git` 的目录列表
- `repoStatus`（第 36 行）：调用 `repoDevices` 再执行 `git status --short --branch`
- `repoPrepare`（第 86 行）：查 `project.created` 事件拿 `repositoryUrl`，确认本地有无后决定是否 clone

`done/cross-device-repo-matrix.md` 遗留的 TODO：`foyer project init/list → foyer repo init/list` 迁移尚未完成，本需求与这条路径有交集（多目录扫描本质上是 `repoDevices` 的能力扩展）。

---

## 决定

### 命名 / 边界

本次 feat 的最小可交付边界：**让 `foyer repo devices` / `foyer repo status` 能感知用户注册的多个扫描根目录**，不触碰 clone 逻辑、不引入 playground 生命周期状态、不合并 `project list → repo list` 迁移。

- 新增命令：`foyer repo roots` — 管理扫描根目录列表（list / add / remove）
- 扩展命令：`foyer repo devices --all-roots` — 跨所有注册根目录聚合扫描结果
- 扩展命令：`foyer repo status --all-roots` — 同上，支持多根聚合状态

命名风格与现有 `foyer repo prepare`、`foyer repo devices` 保持一致，使用 kebab-case 子命令 + `--` 前缀选项。

### 实现要点（复用 vs 新建）

**复用：**

- `repoDevices`（`repo.ts` 第 13-33 行）：接受 `projectsRoot` 参数，保持不动。多根扫描只需在调用侧 `Promise.all` / `Effect.forEach` 多次调用，聚合结果后去重（以 `path` 为 key）
- `repoStatus`（`repo.ts` 第 36-63 行）：同理，调用侧传入多根列表后聚合，函数本身无需改动
- `resolveConfig`（`repo.ts` 第 15-18 行）：沿用现有配置解析逻辑，`projectsRoot` 保持为单值，多根由上层编排

**新建：**

- `scan-roots` 持久化层：读写 `~/.foyer/scan-roots.json`（或与现有 `foyerRoot` 并列）
  - `repoRootsList()`：读取注册的根目录列表，若文件不存在则返回 `[config.projectsRoot]`（向后兼容）
  - `repoRootsAdd(root: string)`：追加一条，去重
  - `repoRootsRemove(root: string)`：移除一条，保留至少一个
- `repoDevicesMulti(input: { roots: string[]; deviceName?: string })`：对 `roots` 做 `Effect.forEach` + 调用 `repoDevices`，合并并去重
- `foyer repo roots list/add/remove` 三条子命令注册到 `index.ts` 的 `repo` 组

### 数据契约

`repoDevices` 现有返回结构：

```ts
{
  device: string;
  repo: string;
  path: string;
}
[];
```

多根扫描结果新增 `scanRoot` 字段，以便调用方区分来源：

```ts
{
  device: string;
  repo: string;
  path: string;
  scanRoot: string;
}
[];
```

`--json` 输出向后兼容：`scanRoot` 为新增字段，不破坏现有消费者（消费者按字段访问，多余字段无害）。

`scan-roots.json` 格式：

```json
{
  "roots": ["~/repo/projects", "~/repos", "~/github"]
}
```

路径存储时保留 `~` 前缀，读取时展开（`os.homedir()`）。

### 前置依赖

- `~/.foyer/` 目录已存在（`foyer doctor` 已检查）
- `repoDevices` 实现不变（本次不重构），兼容旧调用方

## 待用户确认

1. **配置文件位置**：scan roots 存在 `~/.foyer/scan-roots.json` 还是跟随 foyer 数据根（`--foyer-root` 参数所指目录）？后者在多设备场景下可通过 sync 感知其他设备的根目录配置，但引入了跨设备配置同步的额外复杂度。

2. **是否引入 playground 阶段**：tap 提到"playground 的 repo 默认不同步远程"。这个生命周期状态是否属于本次 feat 范围，还是作为单独的后续 feat？引入后需要在 `project.created` 事件（或新事件）里记录 `stage: playground | project`，并影响 `repoPrepare` 的同步行为。

3. **多目录布局规范**：用户是否希望 foyer 给出推荐的目录结构（如统一迁移到单一 `~/repo/projects`），还是完全尊重现状、只做扫描感知而不做目录整合？两者影响 `foyer repo roots add` 是否需要迁移提示。

4. **`foyer project list → foyer repo list` 迁移时机**：`done/cross-device-repo-matrix.md` 留下这个 TODO。本次扩展 `repoDevicesMulti` 后，是否顺带推进这个迁移，还是严格隔离为独立 feat？迁移涉及 `index.ts` 现有 `project list` 命令的 deprecation 处理。

5. **`--all-roots` 默认行为**：`foyer repo devices` 不带 flag 时是否仍只扫描单一 `projectsRoot`（向后兼容），还是直接改为默认扫描所有注册根目录？后者更易用但是 breaking change。
