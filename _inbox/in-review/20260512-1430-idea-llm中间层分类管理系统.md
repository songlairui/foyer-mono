# LLM 中间层分类管理系统

人类的读写不再直接触碰落库的分类信息。增和查的过程加入 LLM 中间层：

- 输入时，人类只管表达自己的分类意图，LLM 对比已有分类，追问是否复用相近项，若不复用则构思一个向量距离足够远的新分类
- 分类集合无限增长，完全由 LLM 掌控，不能全量读取——需要向量存储 + 近邻查询
- 借鉴现有 agent 记忆方案（memoir、Hyper-Extract、graphify、honcho）的 CRUD 理念，应用于分类管理

调研目标：memoir / Hyper-Extract / graphify / honcho，分别存档 docs/wiki/，然后构思分类管理与使用的完整方案。

我想使用LLM把分类这件事情做到极致。传统的就是增删改查; LLM 时代,应该是增的过程与查的过程，加上 LLM 中间层. 人类的读和写，不再直接触碰落库的信息. 因为人类会把分类这个问题搞得很凌乱. 输入的时候,人类只管讲自己的分类.然后 LLM 跟已有分类对比,追问是否复用相近的分类.如果不复用，则构思出一个向量距离不那么近的分类. 一个难点,默认这个分类是无限增加的,完全由llm掌控,所以不能每次让LLM全量读这个分类.应该使用向量的方式存储这个分类再查询相邻关系.再延伸一点儿.现在的各个agent的记忆方案,曾出不穷,意味着大家都感觉有些场景没做好.但,如果我用这些记忆方案的方式处理分类的crud呢,似乎可以变得很强,一定能cover它.比如,我要使用 honcho, graphify, hyper-extract, (https://github.com/zhangfengcdt/memoir; https://github.com/yifanfeng97/Hyper-Extract; https://github.com/safishamsi/graphify; https://github.com/plastic-labs/honcho); 请调研这些方案,阐述它们的设计思想，在管理'分类/tag'上的可借鉴之处.分别存档到 docs/wiki/ 下新建md文件;然后构思我说的 分类 管理和使用的方案
