## 无IO阻塞处理机制设计方案

### 当前流程的阻塞点分析

1. **阶段一的阻塞点**：
   - 需要先读取现有 tag（多个文件读取）
   - 生成 ID、推断 title、type、tags 后才能写入
   - 写入完成才返回确认，用户等待时间长

2. **阶段二的阻塞点**：
   - 需要等待各种分析完成才能决定下一步
   - 多步骤同步执行，缺乏并行处理

### 优化设计方案

#### 一、"快速响应 + 后台处理" 模式

**阶段一：立即响应（< 100ms）**

```
用户输入 → 生成临时 ID → 写入原始内容到队列 → 立即返回确认 ✓
```

- 只做最基础的记录，不加任何分析
- 使用内存队列，不等待文件写入完成
- 立即返回一个临时确认给用户

**阶段二：后台异步处理**

```
后台 worker 从队列取任务 → 分析 → 分类 → 处理 → 更新状态
```

- 在后台进行详细分析和处理
- 不阻塞用户的后续操作
- 通过状态通知机制让用户知道处理进度

#### 二、具体实现细节

**1. 快速响应层**

- 接收用户输入，生成临时 ID（格式：`temp-{timestamp}`）
- 立即返回：`✓ 已捕获，正在处理中... (临时 ID: temp-12345)`
- 将原始内容放入内存队列，无需等待文件写入

**2. 后台处理层**

- Worker 从队列取任务，异步处理
- 处理完成后，更新文件状态，将临时 ID 替换为正式 ID
- 提供状态查询接口：`inbox status {temp-id}`

**3. 状态通知机制**

- 处理完成后，自动发送通知：`✓ 事项已处理完成 (ID: 20260515-0001)`
- 用户也可以主动查询状态

**4. 缓存优化**

- 使用内存缓存保存常用 tag，减少文件读取
- 缓存项目上下文信息，加速分析过程

#### 三、用户体验改进

**1. 连续输入支持**

- 用户可以连续输入多个事项，无需等待前一个处理完成
- 系统会在后台依次处理，给出统一的处理完成通知

**2. 进度可视化**

- 提供简单的进度指示：`已捕获 3 个事项，正在处理第 2 个...`
- 处理完成后给出汇总报告

**3. 容错机制**

- 如果处理失败，保留原始内容，给出修复建议
- 不丢失用户输入的任何信息

#### 四、技术架构示意

```
┌─────────────────┐    ┌──────────────────┐    ┌──────────────────┐
│  用户输入层     │ →  │  快速响应层      │ →  │  后台处理层      │
│  (无阻塞)       │    │  (内存队列)      │    │  (异步分析)      │
└─────────────────┘    └──────────────────┘    └──────────────────┘
                                                  ↓
                                          ┌──────────────────┐
                                          │  文件存储层      │
                                          │  (状态更新)      │
                                          └──────────────────┘
```

### 核心优势

1. **零阻塞体验**：用户输入后立即得到确认，无需等待任何处理
2. **高吞吐量**：可以同时处理多个事项，不影响用户操作
3. **容错性强**：即使后台处理失败，也不会丢失用户输入
4. **可扩展性好**：可以轻松添加更多的后台处理步骤
