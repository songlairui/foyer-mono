## 无IO阻塞机制具体实施示例

### 一、用户交互流程示例

**场景：用户连续输入多个事项**

```
用户：事项一：跟踪的需求，后端有新进展，可以联调了。我正在更新 grpc 协议。

系统：✓ 已捕获 (临时ID: temp-001)，正在后台处理中...

用户：事项二：反馈了一个交互优化问题。将默认自动开始生成的报告，变成，点击开始生成。

系统：✓ 已捕获 (临时ID: temp-002)，正在后台处理中...

用户：还有一个想法...

系统：✓ 已捕获 (临时ID: temp-003)，正在后台处理中...

（2秒后）

系统：✓ 处理完成！
  - 20260515-0001 · chore · GRPC 协议联调更新 → in-progress
  - 20260515-0002 · feat · 报告生成交互优化 → in-progress
  - 20260515-0003 · idea · LLM时代事项管理模式设想 → tapped
```

### 二、处理流程详解

**快速响应阶段（0-100ms）**

```
1. 接收用户输入
2. 生成临时 ID：temp-{timestamp}
3. 将原始内容放入内存队列
4. 立即返回确认：✓ 已捕获 (临时ID: temp-001)
5. 结束响应用户的请求
```

**后台处理阶段（100ms+）**

```
1. Worker 从队列取出 temp-001
2. 读取项目上下文（从缓存或快速扫描）
3. 分析内容：
   - 生成正式 ID：20260515-0001
   - 推断 type：chore
   - 生成 title：GRPC 协议联调更新
   - 提取 tags：["联调", "GRPC", "后端协作"]
4. 润色 body，添加原文详情
5. 写入正式文件：_inbox/in-progress/20260515-0001-xxx-chore-grpc协议联调更新.md
6. 标记 temp-001 为已处理
7. 收集所有已处理事项，准备汇总通知
```

**汇总通知阶段（所有事项处理完成后）**

```
1. 检查队列是否还有待处理事项
2. 如果队列为空，发送汇总通知
3. 列出所有事项的处理结果
```

### 三、具体技术实现

**1. 内存队列设计**

```javascript
// 简单的内存队列实现
const taskQueue = {
  tasks: [],
  callbacks: [],

  add(task) {
    this.tasks.push({
      id: `temp-${Date.now()}`,
      content: task,
      status: "pending",
      createdAt: new Date(),
    });
    this.processNext();
  },

  processNext() {
    if (this.tasks.length === 0) return;

    const task = this.tasks.find((t) => t.status === "pending");
    if (!task) return;

    task.status = "processing";
    this.processTask(task);
  },

  async processTask(task) {
    // 后台处理逻辑
    const result = await analyzeAndSave(task);
    task.status = "completed";
    task.result = result;

    // 检查是否所有任务都完成
    if (this.tasks.every((t) => t.status === "completed")) {
      this.sendSummary();
    } else {
      this.processNext();
    }
  },

  sendSummary() {
    // 发送汇总通知
  },
};
```

**2. 快速响应层实现**

```javascript
// 用户输入处理函数
async function handleUserInput(content) {
  // 立即加入队列
  const tempId = taskQueue.add(content);

  // 立即返回确认，不等待处理
  return `✓ 已捕获 (临时ID: ${tempId})，正在后台处理中...`;
}
```

**3. 后台处理实现**

```javascript
async function analyzeAndSave(task) {
  // 1. 快速分析（使用缓存的上下文）
  const analysis = await fastAnalyze(task.content);

  // 2. 生成正式文件
  const filePath = await saveToFile(analysis);

  // 3. 返回结果
  return {
    id: analysis.id,
    type: analysis.type,
    title: analysis.title,
    status: analysis.status,
    filePath,
  };
}
```

### 四、状态查询机制

**用户可以主动查询处理状态：**

```
用户：inbox status temp-001

系统：
事项 temp-001 处理状态：
- 状态：处理中...
- 已完成：内容分析、tag 提取
- 进行中：文件写入、状态更新

用户：inbox status

系统：
当前处理队列状态：
- 待处理：0 个
- 处理中：1 个 (temp-002)
- 已完成：2 个 (temp-001, temp-003)
```

### 五、容错和恢复机制

**1. 处理失败时的处理**

```
系统：⚠️ 事项 temp-002 处理遇到问题，原始内容已保存
您可以：
- inbox retry temp-002  重新处理
- inbox show temp-002   查看原始内容
- inbox edit temp-002   编辑后重新处理
```

**2. 系统重启后的恢复**

- 队列状态会持久化到临时文件
- 重启后自动恢复未完成的处理
- 保证不丢失任何用户输入

### 六、优势总结

1. **用户体验流畅**：输入后立即得到确认，无等待
2. **高效处理**：后台批量处理，充分利用计算资源
3. **容错性强**：处理失败不影响其他事项，可重试
4. **可观察性好**：提供状态查询，用户随时了解处理进度
